# Процесс нормализации базы данных "Планирование учебного процесса"

## Анализ предметной области

Приложение "Планирование учебного процесса" предназначено для организации учебной деятельности студентов и преподавателей. Основные бизнес-процессы:

- Управление расписанием занятий
- Создание и отслеживание учебных заданий
- Организация заданий по дисциплинам и приоритетам
- Система уведомлений о занятиях и дедлайнах
- Совместная работа над проектами
- Комментирование и обсуждение заданий

## Нормализация до 1NF

### Исходная денормализованная структура (проблемный вариант):
Tasks(task_id, title, description, due_date, priority, status, user_id, username, user_email, project_id, project_name, category_id, category_name, category_description)

### Проблемы:
- Множественные значения в полях категорий
- Повторение данных пользователя в каждой задаче
- Сложность обновления информации о проектах
- Нарушение принципа атомарности атрибутов

### Решения для 1NF:
- Все атрибуты сделаны атомарными
- Множественные значения вынесены в отдельные таблицы
- Созданы отдельные таблицы для связанных сущностей

## Нормализация до 2NF

### Выявленные частичные зависимости:
- username и user_email зависели от user_id, а не от task_id
- project_name зависела от project_id, а не от task_id
- category_name зависела от category_id, а не от task_id

### Решения для 2NF:
- Создана таблица Users с информацией о пользователях
- Создана таблица Projects с информацией о проектах
- Создана таблица Subjects с информацией о дисциплинах
- В Tasks остались только ссылки (FK) на связанные сущности

## Нормализация до 3NF

### Выявленные транзитивные зависимости:
- В первоначальной структуре были зависимости между неключевыми атрибутами
- Атрибуты категорий зависели друг от друга

### Решения для 3NF:
- Каждая таблица содержит только атрибуты, зависящие от первичного ключа
- Созданы таблицы для многих-ко-многим связей (Project_Members, Project_Assignments)
- Вынесены специализированные сущности (Notifications, Comments)

## Финальная нормализованная схема (3NF)

### Таблицы:
1. **Users** - данные пользователей (студенты и преподаватели)
2. **Subjects** - учебные дисциплины
3. **Lessons** - расписание занятий
4. **Assignments** - учебные задания и дедлайны
5. **Projects** - групповые проекты
6. **Project_Members** - связь пользователей с проектами
7. **Project_Assignments** - связь заданий с проектами
8. **Notifications** - система уведомлений
9. **Comments** - комментарии к заданиям

### Преимущества нормализованной схемы:
- Устранена избыточность данных
- Предотвращены аномалии обновления/удаления
- Упрощено поддержание целостности данных
- Гибкость для будущих расширений

## Компромиссы и проектные решения

### Сознательные отступления от строгой нормализации:
- В Assignments хранится updated_at для отслеживания изменений
- В Notifications хранится is_sent для управления состоянием уведомлений

### Индексы для оптимизации:
- Индексы на всех внешних ключах
- Индекс на due_date для быстрого поиска ближайших дедлайнов
- Индексы на часто используемых полях фильтрации (status, priority)
